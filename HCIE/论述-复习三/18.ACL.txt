问题三：
1、使用范围
acl 使用范围更广泛，如下：
标准 acl，可针对 source 匹配路由和数据。
扩展 acl，可针对具体流量，可匹配指定源到目标及协议优先级等具体参数。
二层 acl，匹配二层流量，可针对特定 802.1P、VLAN ID、源目 MAC 地址、二层协议类型进行精确匹配。
ipv6 acl，可针对 ipv6 环境，实现同上述 ipv4 标准和扩展 acl 功能。

而 ip-prefix 只可针对 ipv4、ipv6 环境的网络前缀进行路由层面的控制。

2、路由匹配的灵活性
acl 使用通配符，更便于匹配有规律但不连续的网络。
例如：匹配 192.168.x.0/24 的偶数路由，acl 配置如下：
acl 2000
    rule permit source 192.168.0.0 0.0.254.0

而 ip-prefix 匹配网络范围，其网络号必须连续，无法实现上述需求。

3、路由匹配的精确性
acl 通配符关心网络号，但掩码长度不好精确匹配，可以通过使用前缀列表更加精确的匹配路由的网络前缀范围和掩码长度范围。

例如有以下三条路由，匹配 192.168.1.0/22、192.168.1.0/24 两条路由
192.168.1.0/16
192.168.1.0/22 
192.168.1.0/24

使用 acl：
acl 2000
    rule permit source 192.168.1.0 0
匹配的网络号为 192.168.1.0，不关心掩码长度，所以三条路由都被匹配到，无法区分。

使用 ip-prefix：
ip ip-prefix test permit 192.168.1.0 16 g 20 l 20
匹配的网络号为 192.168.1.0，掩码长度范围为 20 到 24，所以可匹配到 192.168.1.0/22、192.168.1.0/24 两条路由。

ip ip-prefix test permit 192.168.1.0 16
而如果不指定掩码长度范围，则默认掩码长度为前缀长度 16，所以只可匹配到 192.168.1.0/16。

4、对未命中规则的处理方式
acl 无论是 permit 还是 deny，最终执行的动作是由应用 acl 的业务模块决定的，不同业务模块对命中还是未命中的报文处理方式不同。
（1）在匹配路由时，acl 配置规则但未命中，默认动作为 deny，拒绝其它路由通过。
例如：
192.168.1.0/24
192.168.2.0/24

acl 2000
    rule permit source 192.168.1.0 0

isis 1
    filter-policy 2000 export

使用上述 acl 规则，192.168.2.0/24 路由拒绝向其它设备通告。

（2）在匹配流量时，流策略中的 acl 默认动作为 permit，如果 acl 中存在规则但报文未命中规则，该报文可以正常通过。
（3）在 Telnet、Stelnet、SNMP 等业务模块中，acl 默认动作为 deny，如果 acl 中存在规则但报文未命中规则，该报文被拒绝通过。
（4）黑名单模块中的 acl 处理机制和其它业务模块有所不同，无论 acl 规则是 permit 还是 deny，如果命中规则，该报文都会被丢弃。

而 ip-prefix 匹配路由时，默认动作为 deny，存在规则但未命中默认拒绝。
